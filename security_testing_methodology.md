# 安全测试方法论（面向智能体）

> **文档定位**：本文档是指导**智能体（AI Agent）** 执行安全测试的核心方法论，适用于**白盒 / 灰盒**场景——智能体具备代码阅读权限与后台（日志、文件、进程、配置）访问权限，以"代码理解 + 后台观察"为核心手段，不以黑盒扫描或盲打为主。

## 0. 智能体执行原则

以下原则贯穿所有测试动作，优先级高于具体流程。

- **先读码，后动手**：任何输入点测试前，先定位对应后端代码，明确数据流向与可能的危险汇点，再决定如何触发与观察。
- **动作前声明，动作后取证**：每次有意义的测试输入前说明意图与预期观察点；输入后**立即**执行后台搜索/监控，拿到一手证据。
- **证据优先于推断**：所有结论必须有代码片段（文件路径 + 行号）或后台记录（日志片段、文件、进程、网络）作支撑；没有证据时明确说"未观察到"，不要编造、不要脑补利用链。
- **默认低置信度**：仅凭代码推断的结论默认低置信度，主动提出下一步动态验证方案，而不是直接定性。
- **不确定就记录**：观察不到、条件不满足、环境受限时，如实记录缺失项，不要强行给出结论。
- **谨慎动作**：避免破坏性操作（删数据、改配置、大规模请求、覆盖日志）；改动环境前确认授权与回滚手段。

## 1. 测试流程

五阶段框架，"核心方法论"中的观察/验证动作贯穿其中。

### 信息收集
- 梳理代码结构、模块划分、关键接口清单、角色与权限模型。
- 确认依赖组件及版本，对照已知 CVE。
- 明确测试范围与边界。

### 测绘建模
- 从代码出发识别输入点与信任边界，标注用户可控字段与其到达的关键函数。
- 梳理数据流与控制流：参数如何流向文件 / 命令 / SQL / 网络 / 反序列化等危险汇点。
- 结合配置与日志路径，预判可观察的后台信号。

### 动态验证
- 带着代码视角做**针对性触发**——为了验证某条可疑代码路径是否可达、是否按预期处理，而不是盲打 payload。
- **输入前**：预置后台日志、文件、命令、网络监控；明确"要看到什么才算命中"。
- **输入后**：立即按核心方法论取证。

### 定性分析
- 对可疑点做最小化 PoC，排除环境与副作用干扰。
- 结合代码路径分析可复现性、触发条件、影响面与前置依赖。

### 记录上报
- 标注置信度（见第 4 节）。
- 写明代码依据、复现前提、已完成与缺失的验证步骤。
- 证据链需可复核：文件路径 + 行号、日志片段、时间戳、请求响应原文。

## 2. 核心方法论

### 后台日志搜索
对有意义的用户输入（如文件名、特殊字符、长字符串）操作后，根据上下文判断，在后台日志中搜索原始输入，结合代码上下文分析记录情况。

### 后台文件搜索
涉及文件操作后，在后台搜索文件名，确认文件实际落盘目录及记录情况。

### 时间关联分析
- 结合时间窗口分析，避免把非当前操作的日志当成证据。
- 前后端时间可能不完全一致，搜索时留出合理窗口。

### 全量日志监控
如果能确定相关操作对应的后台日志文件路径，全量监控这些日志文件的新增内容——关键信息不一定包含前端输入，仅搜输入字段会漏判。

### 全面操作监控
所有用户操作都主动三问：**是否落盘？是否拉起子进程？是否外联？** 对应监控：
- 文件操作
- 命令执行
- 网络请求

### 关键目录与文件分析
后台涉及的关键目录和文件直接打开分析，必要时执行脚本或命令做深入探查；配置文件与代码结合，常能直接确认可疑行为。

### 代码与后台综合分析
后台观察与代码互为印证：代码解释后台现象，后台验证代码推断的可达性。只有一侧证据时，置信度不应给高。

## 3. 漏洞类别观察点

按漏洞类型列出后台应重点关注的信号（结合代码中对应汇点一起判断）。

- **命令注入**：是否拉起子进程、shell 解释器参数、异常命令行日志。
- **文件上传 / 任意写**：落盘路径、文件名是否保留原始输入、文件权限位。
- **路径穿越**：日志中出现 `..`、或解析后的绝对路径是否越出预期目录。
- **SSRF / 外联**：异常目的 IP/域名、内网地址、非预期的 DNS 解析请求。
- **反序列化 / 模板注入**：类加载异常、反射调用、模板引擎错误栈。
- **SQL 注入**：SQL 错误日志、慢查询、拼接式 SQL 痕迹。
- **越权**：日志中用户 ID 与资源归属不一致的记录。
- **信息泄露**：错误栈、调试接口、备份文件、目录列举命中。

## 4. 漏洞置信度要求

漏洞报告必须标注置信度，分为 **高 / 中 / 低** 三档。仅凭代码分析得出的结论一般置信度不高，需结合动态验证以提升置信度。

### 高置信度
- 已通过动态验证，能稳定复现。
- 具备完整的 PoC（请求/响应、命令执行输出、落盘文件、日志记录等实际证据）。
- 漏洞影响与利用路径明确，无歧义。

### 中置信度
- 代码分析已较为充分，逻辑链完整，但缺少端到端的运行时验证。
- 或动态验证部分成立（例如触发了异常/日志，但未拿到完整利用证据）。
- 需补充动态验证后才能升为高置信度。

### 低置信度
- 仅基于代码分析，未做任何动态验证。
- 依赖较多假设条件（特定配置、特定权限、特定环境）且未确认。
- 作为线索登记，需进一步验证才能定性。

### 验证原则
- 所有仅凭代码分析得出的漏洞，默认视为低置信度，必须补充动态验证。
- 动态验证应在真实/仿真环境中进行，并结合后台日志、文件、命令、网络等多维度证据交叉印证。
- 置信度不足时，明确说明缺失的验证步骤，而非直接上报为确认漏洞。

## 5. 常见陷阱与反干扰

智能体在观察与判断时容易踩的坑，需主动规避。

- **时间不一致**：前后端或多机时钟偏差，搜日志要留窗口，必要时以服务端时间为准。
- **日志脱敏**：敏感字段（密码/手机号/身份证等）在日志中可能被 `****`，直接搜原始输入会漏判；应搜截断前后缀、上下游字段或关联 ID。
- **异步 / 批处理延迟**：日志不一定即时落盘，存在 flush、队列、定时任务延迟，分析时要等一个周期再下结论。
- **缓存命中**：重复请求可能走缓存不到后端代码路径，换参数或加扰动位再测。
- **多副本 / 多节点**：请求入口节点与实际处理节点未必相同，监控要覆盖所有副本，或先定位到实际处理节点。
- **编码差异**：URL / Unicode / Base64 等在不同层解码后形态不同，搜日志时同时搜原文与解码形态。
- **代码与运行态偏差**：代码分支可能因配置、开关、版本、热更新而与实际运行行为不一致，以运行态观察为准。
- **幻觉风险**：智能体特有——不要把"代码里看起来可以利用"当成"已验证可利用"；没看到就是没看到，不要补齐证据链。
