# 安全测试方法论（面向智能体）

> **定位**：本文档指导 AI Agent 在**源码 + 运行时双重可见**的场景下执行安全测试，覆盖代码审计（白盒）与带代码视角的动态验证（灰盒）。核心假设：Agent 可读取业务源码、后台日志、文件系统、进程信息，并能在测试环境对目标发起请求。

## 0. Agent 执行准则

优先级高于任何具体流程，冲突时以准则为准。

1. **代码先行**：接触任何输入点前，先定位对应后端代码路径（入口 → 业务层 → 数据层 → 汇点），画出污点传播链。没有代码依据的怀疑不作为结论。
2. **假设驱动，证据封口**：每次测试用一句话说清——"我假设 X 条件下触发 Y 行为，预期在 Z 位置看到 W 证据"。没看到 W，就不判定 X 成立。
3. **监控前置**：发起动态请求之前先部署观察手段（日志 tail、文件监听、进程/网络监控），而不是事后补救——关键信号不回放。
4. **证据可复核**：所有结论必须附源码引用（`path:line`）和/或运行时证据（日志片段 + 时间戳、文件路径 + 内容片段、进程命令行、抓包摘要）。不得臆测、不得引用未实际看到的内容。
5. **宁缺毋滥**：观察不到、条件不满足、环境受限时，显式记录"未能验证"并给出所需条件，不要为了凑结论降低标准。
6. **最小扰动**：避免大规模请求、数据破坏、配置改写、日志覆盖。敏感操作（写/删/外联）先在低危路径上演练，并确认回滚手段。
7. **幻觉自检**：完成一条发现后自问——"我是在描述代码里的可能性，还是在描述我观察到的事实？" 两者在报告中用不同措辞严格区分。

## 1. 测试流程

五阶段瀑布，允许小循环回溯。

### 1.1 资产与依赖建模
- 梳理仓库：入口模块、路由表、中间件栈、认证/授权组件、序列化点、模板引擎、外部依赖（DB/MQ/RPC/HTTP client）。
- 列出依赖组件及版本，对照已知 CVE。
- 明确测试边界、授权范围、数据红线。

### 1.2 攻击面映射
- 枚举输入源（source）：HTTP 参数 / Header / Cookie / 上传文件 / 消息队列 / 定时任务 / 配置热更新 / 外部回调。
- 识别汇点（sink）：命令、SQL、文件、网络、反序列化、模板/表达式引擎。
- 标注信任边界：哪一层做了校验/转义/白名单，哪一层直接透传。
- 预判可观察信号：每条链路对应哪个日志、哪个目录、哪个外联端点。

### 1.3 假设形成
对每个可疑汇点提出**可证伪**假设，形如：
> "`POST /api/x` 的 `filename` 参数在 `foo.py:42` 未过滤 `..` 直接拼接绝对路径；预期 `../../tmp/poc.txt` 会在 `/tmp/poc.txt` 落盘。"

### 1.4 动态验证
- 预置监控（日志 tail、文件监听、进程/网络观测）。
- 以最小扰动触发，payload 中使用唯一特征串便于检索。
- "输入 → 观察 → 对照代码"三步闭环；**一次只改一个变量**。

### 1.5 定性与上报
- 评估可复现性、前置依赖、影响范围、利用链完整度。
- 标注置信度（见第 5 节）。
- 证据归档：源码引用、PoC、日志片段、落盘样本、时间线。

## 2. 源码审计方法

### 2.1 污点分析（Taint Tracking）
- **Source**：外部不可信输入；包含**二级 source**（DB 中来自用户的字段、缓存、配置中心、消息体）。
- **Sink**：典型清单
  - 命令：`Runtime.exec`、`subprocess.*`、`os.system`、`popen`、shell 内建
  - SQL：字符串拼接的 `execute*`、ORM raw 接口、动态拼接的存储过程
  - 文件：`open`、`File.*`、`Files.*`、解压、软链、`include/require`
  - 网络：HTTP client、socket、DNS、URL 解析
  - 反序列化：`ObjectInputStream.readObject`、`pickle.loads`、`yaml.load`、`unserialize`
  - 模板/表达式：Velocity `evaluate`、SpEL、OGNL、Jinja2 `Template()`、Freemarker
  - 日志/错误回显：堆栈中带用户输入
- **Sanitizer**：要分辨"真正消除风险"与"只改变形态"。白名单 > 黑名单 > 编码；参数化 > 转义。
- **Propagator**：拼接、格式化、序列化、集合赋值；注意污点是否跨越进程、服务、存储边界。

### 2.2 信任边界
- 典型链路：外部 → 网关/WAF → 应用入口 → 业务层 → 数据/下游。
- 每层标注：做了什么校验？绕过什么校验后直达下一层？
- 特别关注**层内隐式信任**：微服务间、同进程模块间是否默认可信。

### 2.3 常见审计切入点
- 配置：硬编码凭证、默认口令、调试开关、不安全默认值。
- 权限注解 / 过滤器链：缺失、顺序错误、白名单接口的例外路径。
- 反射 / 动态加载：类名可控、方法名可控、SPI 机制。
- 异常处理：`catch(Exception){}` 吞异常、`printStackTrace` 回显。
- 并发：共享变量、TOCTOU、幂等缺失、重入。

## 3. 运行时观察方法

代码审计的假设必须在运行时落地。

### 3.1 后台日志搜索
对用户可控的特征输入（文件名、特殊字符、长字符串、带标记 payload）操作后，在后台日志中检索**原始输入及其常见变形**（URL 解码、Unicode、Base64、大小写、截断前后缀）。

### 3.2 文件与落盘追踪
任何文件相关操作后，按文件名或内容片段全盘搜索；关注预期目录之外的落盘。必要时使用 `inotifywait` 监视目标目录。

### 3.3 全量日志监控
对可疑链路涉及的日志文件做 `tail -F` 全量监控——关键信号（异常栈、明文 SQL、外联目标）未必包含前端输入特征，只搜输入字段会漏判。

### 3.4 三问监控（文件 / 命令 / 网络）
每个用户操作主动回答：
- **是否落盘？** — 监控方式：`inotify` / 目录前后 diff / `find -newer`。
- **是否拉起子进程？** — 监控方式：`ps` 轮询 / `auditd execve` / 进程快照 diff。
- **是否外联？** — 监控方式：`ss -tpn` / `tcpdump` / DNS 日志 / 应用层 HTTP client 日志。

### 3.5 时间关联
- 取证以服务端时间为基准；前后端/多机时钟偏差需留窗口。
- 异步与批处理系统的日志可能跨越一个调度周期到达，未命中不等于不触发。

### 3.6 配置与关键文件
关键目录/配置直接打开对照；必要时运行辅助脚本收集快照（版本、启动参数、挂载点、环境变量），用于解释运行时行为。

### 3.7 代码 ↔ 运行时双向印证
- 代码解释现象：为何出现这条日志、为何命中这个分支。
- 运行时验证推断：这条路径真的可达、这个分支真的被执行。
- **单侧证据不得判定为高置信度。**

## 4. 漏洞类别与观察点

每类至少过两道关：代码审计要点 + 运行时观察信号。

| 类别 | 代码审计要点 | 运行时观察信号 |
|---|---|---|
| 命令注入 | shell 字符串拼接、`Runtime.exec(String)`、未 tokenize 参数 | 异常子进程、shell 解释器命令行、`execve` 审计日志 |
| SQL 注入 | 字符串拼接 SQL、raw 接口、二次拼接存储过程 | SQL 错误栈、慢查询、日志中含特征串的完整 SQL |
| 反序列化 | `readObject` / `pickle` / `yaml.load` / unsafe 魔法方法 | 类加载异常、gadget 类加载、进程异常退出 |
| 模板 / 表达式注入 | SSTI、SpEL/OGNL 可控输入、Velocity `evaluate` | 模板引擎错误栈、非预期表达式求值结果 |
| XXE | `DocumentBuilder` 默认配置、DTD/外部实体未禁 | 对外 DNS/HTTP 请求、`file://` 读取痕迹 |
| 文件上传 / 任意写 | 扩展名黑名单、路径拼接、解压未校验 | 落盘路径、文件名是否保留原始输入、权限位异常 |
| 路径穿越 / 包含 | `..` 过滤缺失、软链、Zip Slip | 解析后绝对路径越出预期目录 |
| SSRF | URL 解析后直接请求、协议/地址未限制、重定向跟随 | 外联到内网/元数据地址、异常 DNS 解析 |
| IDOR / 越权 | 直接使用请求中的资源 ID、缺失 ownership 校验 | 日志中 `user_id ≠ resource_owner`、缺 4xx 拦截 |
| 认证 / 会话 | JWT 未校验签名/算法、session 固定、密钥硬编码 | 签名异常日志、重复 session、过期不失效 |
| 竞态 / TOCTOU | 读-改-写非原子、幂等缺失 | 并发下状态不一致、重复扣减/发放 |
| 业务逻辑 | 金额/数量/状态可绕过、流程跳步 | 异常状态转移、非正常金额记录 |
| 加密误用 | 弱算法、ECB、静态 IV、硬编码密钥 | 固定密文前缀、可区分密文模式 |
| 敏感信息泄露 | 日志含明文凭证、堆栈回显、备份文件暴露 | 日志中 token/密码、调试接口响应、`.bak/.swp` 命中 |

## 5. 证据与置信度

### 5.1 证据等级（强 → 弱）
1. **完整 PoC**：输入 + 响应 + 后台落地证据（日志/文件/进程/网络），可稳定复现。
2. **部分运行时证据**：命中预期侧信号（异常栈、异常日志、部分落盘），但未形成完整利用链。
3. **代码逻辑链**：源码推断完整，污点从 source 到 sink 无断点，但无运行时证据。
4. **启发式线索**：片段匹配、模式告警，需进一步定位代码。

### 5.2 置信度分档
- **高**：等级 1 + 稳定复现 + 利用路径无歧义。
- **中**：等级 2，或等级 3 且所有前置条件在当前环境已逐一确认为真。
- **低**：仅等级 3（未动态验证）或等级 4。

**仅凭代码审计得出的结论默认低置信度，必须补充动态验证。**

### 5.3 上报格式
每条发现至少包含：
- 漏洞类型、影响面、CWE 编号（可选）。
- **源码证据**：`path:line` 引用 + 关键片段。
- **运行时证据**：复现步骤、请求/响应、日志片段（含时间戳）、产物文件。
- 前置条件与影响范围。
- 置信度档位及缺失的验证项。

## 6. 常见陷阱与反幻觉

Agent 易犯错的地方，显式对抗。

- **幻觉利用链**：看到危险 sink 就宣布"可利用"——必须验证可达性与污点传递是否无断点。
- **Sanitizer 无效假设**：断言某过滤器"肯定能绕过"但没试——绕过必须有 PoC。
- **死代码误判**：分支在代码里存在，但运行时因配置/开关/特性标志关闭——以运行态为准。
- **框架自动保护被忽略**：ORM 参数化、模板自动转义、CSRF token 常使看似漏洞失效——审计时确认框架行为。
- **日志脱敏掩盖**：敏感字段被 `****`，直接搜原文会漏；应搜截断前后缀、关联 ID、上下游字段。
- **时间/时钟偏差**：搜日志留窗口；跨机以服务端时间为准。
- **异步 / 批处理延迟**：日志 flush、队列消费、定时任务延迟——等一个周期再判空。
- **缓存短路**：重复请求走缓存未到后端——变参数或加扰动位。
- **多副本 / 多节点**：入口节点与处理节点不同——监控覆盖全副本或定位到处理节点。
- **编码形态差异**：URL / Unicode / Base64 / 大小写 / 宽字节 / 规范化——搜原文与解码形态并行。
- **代码与运行态偏差**：热更新、灰度、AB 实验导致差异——以运行态为准。
- **"没看到 = 不存在" 误判**：观察窗口、权限、日志级别不足都会漏；必要时提升日志级别或延长窗口再下结论。
